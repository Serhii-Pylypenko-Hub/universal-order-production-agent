# Code Quality Audit

Дата перевірки: 2026-05-16.

## Висновок

Поточний код відповідає базовим вимогам MVP щодо модульності, ізоляції помилок і валідації користувацьких вводів.

## Що вже закрито

- Бізнес-логіка винесена в окремі сервіси: склад, партії FIFO, виробництво, закупки, каталог, бот, AI.
- Web API має загальний `try/catch` на рівні сервера.
- Системні помилки не мають валити весь веб-кабінет: користувач отримує безпечне повідомлення, а технічні деталі йдуть у `UserFacingErrors` і `DeveloperAlerts`.
- Валідація на бекенді використовує `ValidationError` з масивом помилок.
- Валідація розрізняє:
  - `missing_required` - обов'язкове поле не заповнене;
  - `invalid_value` - поле заповнене, але значення некоректне.
- Фронтенд показує різну підсвітку для порожнього обов'язкового поля і невалідного значення.
- Якщо помилок кілька, користувач бачить усі інструкції українською.
- API `/api/connections` повертає структуровані `validation_errors` з `field`, `type`, `category`, `instruction`.
- Невалідні token/chat ID/API key не зберігаються.

## Модульність

Основні модулі не мають напряму дублювати доступ до таблиць:

- `rowRepository` - спільний доступ до рядків;
- `stockLotService` - партії та FIFO;
- `stockService` - агреговані залишки, резерви, списання;
- `productionService` - виробничі дії;
- `inventoryService` - веб-операції складу;
- `catalogWorkspaceService` - продукти і техкарти;
- `purchaseWorkspaceService` - закупки;
- `botManagementService` - керування ботами;
- `assistantActionService` - AI-команди та дозволи.

## Ізоляція помилок

Помилка в окремій API-дії повертає відповідь тільки для цієї дії. Інші модулі залишаються доступними після оновлення сторінки або повторної дії.

Критичні помилки:

- не показують stack trace користувачу;
- створюють `UserFacingErrors`;
- створюють `DeveloperAlerts`;
- за наявності GitHub-конфігурації можуть створити issue.

## Що перевіряти в наступних PR

- Нові create/update API повинні використовувати `assertValid`.
- Нові форми повинні використовувати `validateFormDetailed`.
- Для кожного нового поля треба мати українську підказку.
- Якщо поле обов'язкове, помилка має бути `missing_required`.
- Якщо поле заповнене неправильно, помилка має бути `invalid_value`.
- Не дублювати FIFO, резерви, списання або перевірку матеріалів у нових місцях.
